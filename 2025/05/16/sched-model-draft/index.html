Pipelined or Not-Pipelined

由于存在长周期指令，比如乘法，除法指令，基本上不可能一个周期完成，所以每个功能单元的占用时间是不同的。

假设目前我们所有的功能单元都是进入一条指令以后就不能再进入第二条指令了，也就是说，当出现如下顺序的汇编时，

如果 mul 指令不能在一个周期完成执行，下一条 mul 就会进入在 MUL 功能单元之前被阻塞住，也就是出现了 pipeline 暂停的行为。

上述对 MUL功能单元的描述，意思就是说，这个功能单元是非流水线化的（Not-Pipelined）。

显然这种 pipeline 是非常慢的，会出现大量因为功能单元被占用导致的流水线暂停，所以这里我们可以引入流水线化的功能单元（Pipelined function unit）。

    ID[ID]
    INT_OP[Integer Unit]
    FP_ADD0[FP Add/Sub 0]
    FP_ADD1[FP Add/Sub 1]
    FP_ADD2[FP Add/Sub 2]
    FP_MUL0[FP Mul 0]
    FP_MUL1[FP Mul 1]
    FP_MUL2[FP Mul 2]
    FP_MUL3[FP Mul 3]
    FP_DIV[FP DIV]
    MUL0[MUL/DIV0]
    MUL1[MUL/DIV1]
    MUL2[MUL/DIV2]
    MEM[MEM]
    WB[WB]

    IF -->| | ID
    ID -->| | MUL0
    ID -->| | FP_DIV
    ID -->| | INT_OP
    ID -->| | FP_ADD0
    ID -->| | FP_MUL0
    INT_OP -->| | MEM
    MUL2 -->| | MEM
    FP_ADD2 -->| | MEM
    FP_MUL3 -->| | MEM
    FP_DIV -->| | MEM
    MEM -->| | WB

    MUL0 --> MUL1 
    MUL1 --> MUL2

    FP_ADD0 --> FP_ADD1
    FP_ADD1 --> FP_ADD2

    FP_MUL0 --> FP_MUL1
    FP_MUL1 --> FP_MUL2
    FP_MUL2 --> FP_MUL3

如上图，FP Mul 单元，Mul/Div 单元以及 FP Add/Sub 单元均已被流水线化。

当再次出现如下顺序的汇编时，流水线将不会因为资源占用导致暂停了。

  1. 大部分功能单元都是流水线化的，但是像 FP Div 这种使用频率较少的功能单元，做成非流水线化的通常情况下也不会导致流水线频繁暂停。做成非流水线化的反而能使 pipeline 更简单。
  2. 这里把功能单元拆成多个 stages 是为了帮助理解，实际的 pipeline 图不会这么画，是否是流水线化的也需要参考对应的白皮书文档。很多 pipeline 信息都是需要从白皮书中获取的。

Real Pipeline

让我们来看一个真实的 pipeline，这是 andes-d45 的 pipeline。

对应的 pipeline 简化图，

flowchart LR
    %% Issue Paths
    Issue0[Issue Path 0]
    Issue1[Issue Path 1]

    %% Pipelines
    ALU0[ALU0]
    ALU1[ALU1]
    LDST[AGU + LD/ST]
    FP[FP Pipeline]
    MULDIV[MUL/DIV Pipeline]
    DSP[DSP Pipeline]

    %% Connections
    Issue0 --> ALU0
    Issue0 --> ALU1
    Issue0 --> LDST
    Issue0 --> FP
    Issue0 --> MULDIV
    Issue0 --> DSP

d45 由于是双发 pipeline，一个周期会发射两条指令，考虑到 alu 指令是最频繁的，为了可以同时双发两条 alu 指令，它这里设置了两条 ALU通路，也就是说，当出现下面两条汇编指令同时发射时，它们可以分别占用两条 ALU通路，而不会导致流水线暂停。（可以想到，如果只有一条 ALU 通路的话，那么这里两条 add 指令就没法同时发射了，因为会有资源冲突，流水线会暂停）

d45 其他的部分并没有什么特殊的地方，一个 LD/ST 功能单元，一个 MUL/DIV 功能单元，一个 DSP 功能单元…

另一个是 sifive-p600 的 pipeline，它是一个乱序 4 发射的 pipeline，

从图上看，可能会感觉功能单元这么多又这么复杂，但其实不然，这是一个很简单的 pipeline，我们先将乱序以及一些Cache，prefetchr，分支预测部分剖除，会得到下面这样的一个 pipeline 图。

flowchart TD

    Issue0[Issue Path 0]
    Issue1[Issue Path 1]
    Issue2[Issue Path 2]
    Issue3[Issue Path 3]

    %% ALU Pipelines

    ALU_ONLY[ALU]

    %% LD/SD Pipelines
    LDST0[AGU + LD/ST]
    LDST1[AGU + LD/ST]

    FP0[FP: FMAC + FADD + FMUL + I2F]
    FP1[FP: FMAC + FADD + FMUL + FDIV]

    VEC0[Vector: ALU + DIV]
    VEC1[Vector: ALU + V2S]

    %% Connections from Issue paths
    Issue0 --> ALU_FULL
    Issue0 --> ALU_ONLY
    Issue0 --> BR0
    Issue0 --> BR1

    Issue0 --> LDST0
    Issue0 --> LDST1

    Issue0 --> FP0
    Issue0 --> FP1

    Issue0 --> VEC0
    Issue0 --> VEC1

功能单元的名字和功能不重要，对调度来说，都只是功能单元。
